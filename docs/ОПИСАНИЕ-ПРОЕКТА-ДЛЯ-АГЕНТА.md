# Полное описание проекта cda-medical-card для нового агента

## 1. Назначение проекта

**cda-medical-card** — веб-приложение «Оркестратор актива» для визуализации и управления данными цифрового двойника актива (ЦДА) в нефтяной тематике. Включает: карту объектов, розы ветров, 3D-гиперкуб рычагов влияния, доску планирования (BPM), конфигуратор систем (схема/код в стиле workflow), результаты и дашборды, ИИ-помощника с голосом и сценариями.

- **Репо:** по факту используется `cda-medical-card`, деплой/упоминания — `axaleiger/cda` или `Axaleiger/syundyukov-alexander-cda`.
- **Стек:** React 18, Vite 5, React Three Fiber + Three.js (3D), recharts, react-simple-maps, xlsx, zustand. Только ESM (`"type": "module"`).

## 2. Точка входа и структура приложения

- **Точка входа:** `src/main.jsx` рендерит `<App />` в `#root`.
- **Главный компонент:** `src/App.jsx` — хранит глобальное состояние (вкладки, BPM-команды, flowCode, ИИ-режим и т.д.) и рендерит либо упрощённый layout (BPM на весь экран), либо полный layout с сайдбаром и вкладками.

### 2.1 Ранние выходы (early return) в App.jsx

Порядок проверок в `App` важен:

1. **cdPageNode** — если задан, рендерится только `CDPage` (страница узла ЦД).
2. **showConfiguratorDoc** — рендерится только `ConfiguratorDocPage`.
3. **showBpm** — рендерится упрощённый layout: только `BPMBoard` + `AIAssistantWidget` + панель мышления. Остальные вкладки (Главная, Планирование в сайдбаре, Конфигуратор и т.д.) в этом режиме **не показываются**. Переключение вкладок через `setActiveTab` при `showBpm === true` не меняет видимый контент, пока не вызвать `setShowBpm(false)`.
4. Иначе рендерится **основной layout** с сайдбаром вкладок и контентом по `activeTab`.

Поэтому сценарий «создать кейс» в начале вызывает **setShowBpm(false)**, чтобы приложение перешло в основной layout и реально показало вкладку «Главная страница» и затем «Планирование».

### 2.2 Вкладки (TABS)

- `face` — **Главная страница**: карта России, розы ветров, секция гиперкуба, Cash Flow, жизненный цикл.
- `scenarios` — Список сценариев (фильтры по этапам, клик по сценарию открывает Планирование).
- `planning` — **Планирование**: доска BPM (этапы, карточки). Lazy: `const BPMBoard = lazy(() => import('./components/BPMBoard'))`.
- `ontology` — **Конфигуратор систем**: вкладки «Схема» и «Код», синхрон с Mermaid-кодом и доской.
- `results` — Результаты (дашборды, метрики).
- `admin` — Администрирование (подвкладки: роли, каталог, заявки и т.д.).

URL: вкладка синхронизирована с `window.location.hash` (`#face`, `#planning`, `#ontology` и т.д.). Приложение читает hash при загрузке и при `hashchange`. Отдельно обрабатываются `#/service/...` (страница сервиса внутри Планирования) и `#admin-<sub>`.

## 3. Ключевое состояние в App.jsx

Краткий список состояния, которое нужно учитывать при доработках:

- **activeTab** — текущая вкладка.
- **showBpm** — режим «только BPM» (например, вход по ссылке с `?bpm=1`).
- **bpmCommand** — объект команды для BPM: `{ scenarioId, params }`. Когда BPMBoard обрабатывает команду, он вызывает **onBpmCommandConsumed(opts)**. В `onBpmCommandConsumed` вызываются: `setBpmCommand(null)`, при `opts?.flowCode` — `setFlowCode(opts.flowCode)`, при `opts?.switchToOntology !== false` — `setActiveTab('ontology')`, `setOpenConfiguratorFromPlanning(true)`.
- **flowCode** — строка Mermaid (flowchart LR). Источник правды для «Код» и «Схема» в Конфигураторе. Обновляется при изменении доски через `onBoardChange` → `bpmToMermaid(stages, tasks)` и при завершении ИИ-кейса через `onBpmCommandConsumed({ flowCode })`.
- **openConfiguratorFromPlanning** — флаг «открыли конфигуратор из планирования». OntologyTab при `openFromPlanning && flowCode` парсит `flowCode` в узлы/рёбра и строит схему, затем вызывает `onOpenFromPlanningConsumed()`.
- **hypercubeCaseIntro** — флаг подсветки гиперкуба при сценарии «создать кейс»: показ Главной, раскрытие воронки, яркая обводка секции.
- **bpmStages, bpmTasks** — последнее состояние доски, передаётся в BPMBoard как **initialStages / initialTasks**, чтобы при переключении вкладок и обратно доска не теряла данные.
- **aiMode** — включён ли ИИ-режим (кнопка в шапке). От него видимость виджета ИИ-помощника.
- **thinkingSteps, thinkingCurrentMessage, thinkingPaused, thinkingPanelOpen** — состояние панели «Режим мышления» и шагов.
- **resultsDashboardFocus** — фокус на метрике на вкладке Результаты (NPV, запасы и т.д.).
- **configuratorNodeCommand** — команда добавить узел на схему (например, из ИИ: «добавь блок X»).
- **selectedAssetId, selectedScenarioName, bpmHighlight, servicePageName, adminSubTab** — выбор актива на карте, сценария, подсветка карточки, страница сервиса, подвкладка админки.

## 4. ИИ-помощник: цепочка от фразы до сценария

### 4.1 Виджет и контекст

- **Компонент:** `src/components/AIAssistantWidget.jsx`.
- Рендерится в двух местах: в упрощённом layout (при `showBpm`) и в основном layout. В обоих случаях ему передаются: **setActiveTab, setBpmCommand, setConfiguratorNodeCommand, setResultsDashboardFocus, setHypercubeCaseIntro, setShowBpm**, а также **onBpmCommandConsumedRef**, **addThinkingStep**, **setThinkingSteps**, **isPaused** и т.д.
- Пользователь вводит текст (или голос через микрофон). По нажатию отправки вызывается **classifyIntent(text)** из `src/lib/intentClassifier.js`. Результат: `{ scenarioId, confidence, topic?, metric? }`.

### 4.2 Классификация намерений (intentClassifier.js)

- Без внешнего API. Ключевые сценарии и фразы:
  - **createPlanningCase** — «сформируй кейс», «создай кейс», «сделай кейс», «кейс по …», «кейс на тему …» и т.п. Из текста извлекается **topic** (тема кейса).
  - **focusMetric / focusNPV** — «упор на NPV», «фокус на запасы», «дашборд по добыче» и т.д. Возвращается **metric**.
  - **buildFullProject** — «полный проект», «от идеи до расчёта».
  - **analyzeRisks** — «риски», «анализ рисков», «матрица рисков».
  - **generateCashflow** — «cashflow», «денежные потоки».
  - **addConfiguratorNode** — «добавь блок …».
  - **appendPlanningCard** — «добавь ещё карточку» и т.п. (используется lastTopicRef от предыдущего кейса).
  - **addPlanningStage** — «добавь стадию/этап …».

Если confidence ≥ 0.95 (или порог из логики), вызывается **runScenario(scenarioId, ctx, topicOrMetric)** из `src/lib/scenarioExecutors.js`. Контекст **ctx** содержит переданные из App функции: setActiveTab, setBpmCommand, setHypercubeCaseIntro, setShowBpm, setConfiguratorNodeCommand, setResultsDashboardFocus, addThinkingStep, isPaused.

### 4.3 Исполнители сценариев (scenarioExecutors.js)

- **runScenario(scenarioId, context, topicOrMetric)** — выбирает функцию по `scenarioId` и вызывает её: `fn(context, topicOrMetric)`.
- Важно для «создать кейс»:
  - **createPlanningCase(ctx, topic)**:
    1. addThinkingStep('Начинаю с главной страницы…').
    2. **setShowBpm(false)** — выход из режима «только BPM», чтобы отрисовался основной layout.
    3. **setActiveTab('face')** — вкладка «Главная страница».
    4. delay(350).
    5. addThinkingStep('Раскрываю дерево этапов и связей в гиперкубе…').
    6. **setHypercubeCaseIntro(true)** — гиперкуб раскрывает воронку и подсвечивается (Hypercube3D получает `highlightCaseTree`).
    7. delay(CASE_INTRO_DURATION_MS) (~2.5 с).
    8. setHypercubeCaseIntro(false).
    9. addThinkingStep('Переход к планированию…'), **setActiveTab('planning')**.
    10. delay(PLANNING_MOUNT_DELAY_MS) (900 ms), чтобы BPMBoard успел смонтироваться.
    11. Генерация шагов: **generateSmartSteps(topicLabel)** из `llmStepGenerator.js` (Groq API или fallback). При ошибке — **DEFAULT_STEPS**.
    12. **setBpmCommand({ scenarioId: 'createPlanningCase', params: { topic: topicLabel, steps } })**.
    13. Цикл addThinkingStep по каждому шагу с задержкой.

- Остальные сценарии: **focusMetric**, **buildFullProject**, **analyzeRisks**, **generateCashflow**, **addConfiguratorNode**, **appendPlanningCard**, **addPlanningStage** — переключают вкладки и/или вызывают setBpmCommand / setConfiguratorNodeCommand / setResultsDashboardFocus.

## 5. Гиперкуб на Главной странице

- **Компонент:** `src/components/Hypercube3D.jsx`.
- На главной рендерится в секции с классом `hypercube-section`; при **hypercubeCaseIntro** секции добавляется класс `hypercube-case-intro` (стили в App.css), а Hypercube3D получает **highlightCaseTree={hypercubeCaseIntro}**.
- При `highlightCaseTree === true` Hypercube3D в useEffect выставляет **selectedVariantId(0)** — открывается воронка уровней (ЦД программ, ЦД объекта, Сервисы, Микросервисы, Функции). Контейнеру задаётся класс `hypercube-highlight-case-tree` (подсветка в Hypercube3D.css).

## 6. Доска планирования (BPMBoard)

- **Компонент:** `src/components/BPMBoard.jsx` (большой файл, ~2300 строк).
- **Пропсы (главные для сценариев):**  
  **bpmCommand**, **onBpmCommandConsumed**, **initialStages**, **initialTasks**, **highlightCardName**, **onBoardChange**, **aiMode**, **setAiMode**, **onClose**, и др.

### 6.1 Обработка bpmCommand

В **useEffect** от `[bpmCommand, onBpmCommandConsumed]`:

- Если `!bpmCommand?.scenarioId || typeof onBpmCommandConsumed !== 'function'` — выход.
- **createPlanningCase:**  
  Берёт `topic` и `steps` из `bpmCommand.params`. Этапы задаются из пресета (например getInitialBoard('hantos')). Формируется список названий карточек (первая — «Кейс: …», остальные из `steps`). Выполняется:
  - setStages(stagesList), setTasks(initialTasks), setAiCaseStageNames, setAiCaseCardIds и т.д.
  - Для каждой карточки — setTimeout с задержкой, внутри setTasks добавляется задача в нужный этап. В **последнем** setTimeout вызывается **onBpmCommandConsumed({ flowCode: bpmToMermaid(stagesList, fullTasks) })**. Так в App уходит актуальный Mermaid и выполняется переход на Конфигуратор.
- **analyzeRisks** — добавляется этап «Риски» и карточки (идентификация рисков, оценка, матрица, план реагирования), затем onBpmCommandConsumed().
- **generateCashflow** — этап «Финансы» и карточки (Cashflow, прогноз выручки, NPV), затем onBpmCommandConsumed().
- **appendPlanningCard** — одна карточка в первый этап; onBpmCommandConsumed({ switchToOntology: false }).
- **addPlanningStage** — добавление этапа по имени; onBpmCommandConsumed({ switchToOntology: false }).

Итог для «создать кейс»: на доске появляются этапы и карточки, после последней карточки вызывается **onBpmCommandConsumed({ flowCode })**, в App обновляется **flowCode**, ставится **activeTab = 'ontology'** и **openConfiguratorFromPlanning = true**.

### 6.2 Синхронизация с конфигуратором при ручном редактировании

- **onBoardChange(stages, tasks)** вызывается при изменении stages/tasks. В App в handleBoardChange делается **setFlowCode(bpmToMermaid(stages, tasks))** и **setBpmStages/S setBpmTasks**, так что при переключении на Конфигуратор схема строится по текущей доске.

### 6.3 ИИ-автопредложения на доске

- Кнопка «ИИ-автопредложения» включает **aiSuggestionsOn**. При этом добавляются два ИИ-этапа (колонки) и три ИИ-карточки в обычных этапах (через **aiSuggestionCardPlacements**). ИИ-карточки вне этапов имеют id вида **ai-inline-0/1/2** (отдельное состояние от карточек внутри ИИ-этапов). Карточки внутри ИИ-этапов хранятся в **tasks[AI_STAGE_NAME]** и **tasks[AI_STAGE_NAME_2]** с флагом **isAiStatic**. Перетаскивание и «Согласовать» работают и для ИИ-карточек, и для ИИ-этапов. Подробности — в BPMBoard.jsx (поиск по aiSuggestionsOn, aiSuggestionCardPlacements, handleStageDropWithAi, getStageDisplayItems).

## 7. Конфигуратор систем (OntologyTab)

- **Компонент:** `src/components/OntologyTab.jsx`.
- Режимы: **Схема** (ConfiguratorCanvas) и **Код** (textarea с Mermaid).
- **flowCode** и **openFromPlanning** приходят из App. При **openFromPlanning && flowCode** в useEffect:
  - setMode('schema'),
  - **mermaidToSchema(flowCode)** из `src/lib/schemaToMermaid.js` даёт узлы и рёбра,
  - узлы центрируются (**centerNodesInField**),
  - setSchemaNodes / setSchemaEdges,
  - вызывается **fitViewRef.current** (центрирование канваса) с таймаутами 50/200/400 ms,
  - затем **onOpenFromPlanningConsumed()** (сброс флага в App).

Схема рисуется в **ConfiguratorCanvas** (канвас в стиле n8n: pan/zoom, узлы, рёбра, создание связей по клику на два узла, удаление связей, ПКМ — панель редактирования и т.д.). Код из редактора можно «Применить к схеме» — парсится через mermaidToSchema и обновляются schemaNodes/schemaEdges.

### 7.1 Связка Планирование → Схема

- При создании ИИ-кейса: BPMBoard в onBpmCommandConsumed отдаёт **flowCode**. App делает setFlowCode, setActiveTab('ontology'), setOpenConfiguratorFromPlanning(true). OntologyTab при монтировании/обновлении видит openFromPlanning и flowCode и строит схему по Mermaid. Без вызова setShowBpm(false) в начале сценария пользователь мог бы остаться в режиме «только BPM», и переход на Конфигуратор не был бы виден.

## 8. Ключевые библиотеки и данные

- **src/lib/bpmToMermaid.js** — преобразование stages + tasks в строку Mermaid flowchart LR (подграфы по этапам, узлы-карточки, рёбра между карточками и этапами).
- **src/lib/schemaToMermaid.js** — конвертация узлов/рёбер схемы в Mermaid и обратно (**mermaidToSchema** — парсинг Mermaid в nodes/edges для канваса).
- **src/lib/llmStepGenerator.js** — **generateSmartSteps(topic)** — запрос к Groq API для генерации названий этапов/карточек; при отсутствии ключа или ошибке возвращается статический FALLBACK_STEPS. Ключ: `import.meta.env.VITE_GROQ_API_KEY`.
- **src/data/bpmData.js** — getInitialBoard(boardId), PERSONNEL, SYSTEMS_LIST, начальные этапы/задачи по пресетам (hantos, nng, mgn).
- **src/data/bpmExcel.js** — загрузка/выгрузка доски в Excel, шаблон.
- **src/data/rosesData.js** — PRODUCTION_STAGES, OBJECTS_BY_STAGE для роз ветров.
- **src/data/scenariosData.js** — SCENARIO_STAGE_FILTERS и данные сценариев.
- **src/data/assetStatus.js** — статусы активов (для карты/стикера).
- **src/data/funnelEntities.js**, **loadFunnelFromExcel.js** — данные для уровней воронки гиперкуба.

## 10. Известные проблемы и недавние исправления

- **Переход к гиперкубу и создание кейса не срабатывали:** из-за режима **showBpm** (режим «только BPM») вкладка «Главная» и контент по activeTab не рендерились. Исправление: в createPlanningCase в начале вызывать **setShowBpm(false)** и передавать **setShowBpm** в AIAssistantWidget и в ctx.
- **Схема не строилась после ИИ-кейса:** нужно, чтобы BPMBoard гарантированно вызывал onBpmCommandConsumed с **flowCode** после добавления всех карточек; в App при opts.flowCode вызывается setFlowCode перед setActiveTab('ontology') и setOpenConfiguratorFromPlanning(true). Также добавлена задержка перед setBpmCommand (PLANNING_MOUNT_DELAY_MS), чтобы BPMBoard успел смонтироваться.
- **ИИ-карточки и ИИ-этапы:** разделены id для карточек «вне» этапов (ai-inline-*) и «внутри» ИИ-этапов (ai-static-*), чтобы не дублировать состояние (раскрытие секций, «Согласовать»). Оба ИИ-этапа заполнены статическими карточками; перетаскивание внутри этапов и между этапами реализовано через handleStageDropWithAi и хранение статических карточек в tasks с isAiStatic.
- **Обмен двух ИИ-карточек местами** вызывал падение из-за дублирования index в aiSuggestionCardPlacements и появления undefined в getStageDisplayItems. Исправлено: reorder по displayIndex, хранение fromIndex в ref при перетаскивании, защита в getStageDisplayItems (не пушить task, если list[tIdx] undefined).
