# Фронтенд: слой данных и переход на API бэкенда

Документ в `Sin/docs/`. Код: `syundyukov-alexander-cda/frontend/`. Сверка с `BACKEND_API_AND_LOGIC.md` и реализованными маршрутами FastAPI.

**Позиция по миграции:** промежуточный режим «static + флаг» в **продакшене не нужен**. Цель — **сразу** перевести приложение с захардкоженных данных, JSON в репозиториях и сценариев, подгружаемых из Excel в браузере, на **единственный источник истины — HTTP API бэкенда**. Ниже — как это согласовать с уже существующей идеей «портов» (репозиториев).

---

## 1. Как устроено сейчас (исходная точка)

### 1.1 Порты и реализации

Доступ к данным идёт через объекты-репозитории с фиксированным набором методов. Контракты — JSDoc в `core/data/repositories/contracts/repositoryContracts.js` (`AppRepositories` и отдельные `*Repository`).

Сейчас единственная «боевая» реализация — **`static/`**: `createStatic*Repository()` читает `core/data/static/*.js`, JSON, генераторы. Сборка — `createStaticRepositories.js`, composition root — `registry.js` (`getRepositories` / `setRepositories`).

### 1.2 React

`DataRepositoriesProvider` + `useRepositories()`; корень — `App.jsx`. Отдельно **`appStore.js`** вызывает `getRepositories()` при инициализации вне React — при переходе на API нужна **асинхронная инициализация** или загрузка фильтров этапов после первого запроса к API.

### 1.3 Прокси

`vite.config.js`: `/api` → бэкенд, префикс `/api` снимается → клиент ходит на `fetch('/api/v1/...')`.

### 1.4 Что уходит

- **Захардкоженные списки** (`PERSONNEL`, генерация сценариев, пресеты BPM из `bpmData.js` как источник истины) — заменяются ответами API (`/users`, `/scenarios`, данные доски из `/planning/cases/...`).
- **Скачивание / загрузка Excel как основной способ получить доску** — в целевом виде доска **читается и сохраняется через API**; файлы Excel остаются максимум как **экспорт/импорт** (если бэкенд добавит `POST .../import-excel`), но не как обязательный путь для работы UI.

## 2. Целевое состояние (без промежуточного гибрида)

| Принцип | Смысл |
|---------|--------|
| **Один runtime-источник** | В собранном приложении `registry` отдаёт **только** набор HTTP-репозиториев (`createHttpRepositories()`). |
| **Нет ветки static в проде** | `createStaticRepositories` **не подключается** в production bundle как путь по умолчанию. |
| **Static только вне приложения** | Старые JSON/генераторы допустимы **в unit-тестах** (моки `fetch`), **fixtures** или временных скриптах сравнения — не как слой, через который ходят экраны. |
| **Бэкенд опережает или синхронно** | Если экрану нужен домен (карта, воронка, lifecycle), а эндпоинта ещё нет — сначала **добавляется API и сид**, затем фронт; обхода «оставить static для этого модуля» в проде нет. |

Исключение по смыслу не «гибрид», а **другой composition root** только в тестах: `setRepositories(mockRepositories)` или `msw` без подключения `static/` из прод-кода.

## 3. Предлагаемая структура каталогов

```
frontend/src/core/data/
├── repositories/
│   ├── contracts/
│   │   └── repositoryContracts.js    # все методы под REST, async где нужен сеть
│   ├── http/
│   │   ├── httpClient.js             # fetch, базовый URL, ошибки, позже auth
│   │   ├── mappers/                  # DTO → форма, удобная для существующих компонентов (при необходимости)
│   │   ├── httpScenariosRepository.js
│   │   ├── httpPlanningRepository.js
│   │   ├── httpUsersRepository.js    # me + catalog
│   │   ├── httpMapGlobeRepository.js # когда появятся GET /map, /assets/.../chains
│   │   └── …
│   ├── createHttpRepositories.js     # единственная сборка для приложения
│   └── registry.js                   # default: createHttpRepositories()
├── static/                           # удалить из импортов приложения после миграции;
│                                     # или вынести в tests/fixtures (не в core/data/runtime)
└── …
```

После миграции папку **`repositories/static/`** и **`createStaticRepositories.js`** имеет смысл **удалить** из репозитория или оставить только в ветке/архиве, чтобы не было соблазна снова подключить.

## 4. Контракты (`repositoryContracts.js`)

Расширить и перевести на **асинхронные** методы там, где данные с сети:

| Порт | REST | Заметки |
|------|------|--------|
| `UsersRepository` | `GET /api/v1/me`, `GET /api/v1/users` | шапка, выбор исполнителя |
| `ScenariosRepository` | `GET /api/v1/scenarios`, `GET .../scenarios/{id}` | список/карточка; маппинг полей UI ↔ DTO |
| `PlanningRepository` | `GET /api/v1/planning/cases`, `GET .../cases/{id}` | `board` из API вместо Excel |
| `AssetsRepository` | `GET /api/v1/assets`, `GET .../assets/{id}` | выбор актива, привязка сценариев |
| `TaxonomyRepository` или поля в Roses | `GET /api/v1/taxonomy/production-stages` | фильтры этапов вместо строк из `rosesData` |
| Остальное (розы по объектам, воронка, lifecycle, сравнение сценариев) | отдельные эндпоинты по мере готовности бэкенда | до появления API — экран не отдаёт данных из static, а **пустое состояние / заглушка** или скрытие блока |

## 5. Слой HTTP

- **`httpClient`**: единая точка `fetch('/api/v1/...')`, разбор ошибок, единый формат ошибки для UI.
- **Мапперы** только там, где модель API расходится с тем, что ждут таблицы/виджеты (например, переход с `stageType` строкой на `productionStageId` UUID — либо меняем компоненты под API, либо тонкий mapper).
- **Загрузка и кэш:** по желанию **TanStack Query** сразу для списков и кейсов планирования (инвалидация после мутаций), либо обёртки в репозиториях с кэшем вручную — но **источник данных всё равно только API**.

## 6. Порядок работ (прямой переход, без «половины на static»)

1. **Дополнить бэкенд** минимальным набором для первого полноценного потока (актив → сценарии → планирование → доска), если чего-то не хватает — без дублирования бизнес-данных на фронте.
2. **Реализовать HTTP-репозитории** и подключить `createHttpRepositories()` в `registry` как единственный default.
3. **Переписать потребителей** (`useScenariosData`, `BPMBoard`, шапка, стор): async-загрузка, UUID/id из API, доска из `GET .../planning/cases/{id}`.
4. **Удалить** импорты `static` репозиториев, генераторы сценариев из runtime-пути, убрать автозагрузку Excel как основной сценарий (оставить опциональный импорт, если появится эндпоинт).
5. **Тесты:** моки `fetch` / MSW с JSON из бывших static-файлов — не прод-код.
